# Como eu contribuo

Olá! 👋

Este não é um guia sobre como contribuir com os meus projetos. É uma forma de mostrar como procuro colaborar quando participo dos projetos de outras pessoas.

Cada mantenedor possui sua forma de trabalhar, e acredito que contribuir também significa respeitar essa forma.

## Antes de escrever qualquer código

Procuro entender o projeto antes de sugerir mudanças.

Leio a documentação, observo a estrutura do código, analiso o estilo utilizado e tento compreender as decisões já tomadas. Meu objetivo não é adaptar o projeto ao meu jeito de desenvolver, mas adaptar minha contribuição ao projeto.

Sempre que uma alteração for significativa, prefiro discutir a ideia antes de implementá-la.

## Respeitando o projeto

Quando contribuo, procuro preservar a identidade do projeto.

Isso significa:

* seguir o padrão de código já existente;
* respeitar a arquitetura adotada;
* utilizar as ferramentas configuradas pelo projeto;
* evitar alterações desnecessárias;
* manter cada contribuição simples e objetiva.

Acredito que consistência é mais importante do que preferência pessoal.

## Commits

Gosto de manter um histórico de commits limpo e descritivo.

Sempre que possível, cada commit representa uma única alteração lógica e possui uma mensagem clara sobre o que foi feito.

Também procuro evitar commits com alterações sem relação entre si.

## Pull Requests

Quando abro um Pull Request, tento facilitar ao máximo a revisão.

Procuro:

* manter o escopo pequeno;
* explicar claramente a motivação da mudança;
* responder comentários com respeito;
* realizar os ajustes solicitados quando fizerem sentido.

Entendo que a decisão final sempre pertence aos mantenedores.

## Revisões

Recebo revisões como parte natural do desenvolvimento.

Não interpreto comentários técnicos como críticas pessoais.

Se alguém encontrar uma solução melhor do que a minha, fico feliz em aprender com ela.

## Comunicação

Acredito que uma boa comunicação evita a maior parte dos problemas.

Por isso procuro:

* ser respeitoso;
* fazer perguntas antes de assumir algo;
* explicar minhas decisões quando necessário;
* ouvir diferentes pontos de vista.

Discordâncias fazem parte do desenvolvimento de software. O importante é que elas permaneçam técnicas e respeitosas.

## Aprendizado contínuo

Cada projeto tem algo para ensinar.

Contribuir para projetos de outras pessoas é uma das formas que encontrei de evoluir como desenvolvedor, conhecer novas abordagens e compartilhar aquilo que aprendi ao longo do caminho.

Espero que toda contribuição que eu fizer deixe o projeto um pouco melhor do que o encontrei.
